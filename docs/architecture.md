# 架构：CRM

## 系统边界

- 本仓库承接的是前台 CRM，负责线索进入、联系人建档、客户归属、跟进沉淀、公海规则和轻量商机。
- 本仓库不承接 ERP 的财务、库存、采购、发货、开票和复杂履约逻辑。
- 成交前的客户关系状态以 CRM 为事实源，成交后的订单与财务状态以 ERP 为事实源。
- AI 背调、培训助手、智能话术和自动外呼属于第二阶段扩展能力，不纳入首版核心架构。
- 工程实现不再从零搭建，而是采用 Twenty 作为上游基线，并在其基础上做本地裁剪与扩展。

## 建议的仓库结构

- `docs/` 用于简报、架构和运维说明
- `specs/` 用于长期维护的实现规格
- `<upstream twenty workspace>` 作为后续导入的主代码基线
- `<local customization layer>` 用于本仓库的领域约束、配置和定制模块
- `docs/` 和 `specs/` 继续作为本仓库的事实来源

## 技术栈

- 上游基线：Twenty
- 技术实现、目录结构、构建脚本和测试约定优先遵循 Twenty 的上游工程
- 本仓库不再维护此前自定义的 Next.js + NestJS + Drizzle 绿地技术方案
- 本地新增能力必须首先判断能否复用 Twenty 现有模块、对象模型和工作流能力

## 选型理由

- 选择 Twenty，是因为它已经是成熟开源 CRM，不需要本仓库再从零发明客户对象、权限、工作流和 CRM 模块边界。
- 选择上游基线而不是自研框架，是为了让 Codex 后续开发直接遵循现成代码结构、命名方式和实现模式。
- 选择基于上游裁剪，而不是继续推进绿地框架，是为了降低前期工程设计成本并减少长期规范分叉。

## 本地裁剪重点

- 保留：客户、联系人、权限、活动流、工作流、列表页与详情页等 CRM 通用能力。
- 定制：公海规则、未建档联系人归属策略、轻量商机、渠道接入优先级、ERP 映射字段。
- 延后：AI 自动外呼、背调、复杂营销自动化以及超出前台 CRM 边界的业务模块。

## 第一批字段落地策略

- 在当前导入分支中，`Company` / `Person` 的标准对象源码入口不完整，因此第一批 CRM 字段不直接修改缺失的标准对象源码。
- 第一批 CRM 字段通过 Twenty 现有的 `field-metadata` custom field 流落地到 `Company` / `Person`。
- 列表默认可见列不直接来自对象字段元数据，而是来自 `CoreView(key=INDEX).viewFields`；因此默认列落地采用“先建字段，再 create/update INDEX viewField”的两段式。
- 默认详情页 `Home` 分组优先复用 Twenty 现有 `Fields` widget 配置，不另造对象详情页；当前通过默认 page layout 常量提供 Company/Person 分组配置，再在前端把字段名解析为真实 metadata id。
- 在默认布局尚未持久化到 page layout metadata 之前，`Company` / `Person` 的首版详情页分组以默认前端 layout 常量为事实来源。

## 核心领域对象

- 客户：以公司或组织为主体，承载归属、公海状态、最近获得时间、最近跟进时间、最近成交时间等关键字段。
- 联系人：挂在客户之下，也可在未建档阶段独立存在，至少支持邮箱、手机号、 WhatsApp、WeChat、阿里旺旺、Facebook ID 等触点字段。
- 跟进事件：记录发/收 WhatsApp、发/收邮件、网站聊天、手工记录、新建商机等行为，并驱动时间字段更新。
- 商机：轻量状态对象，首版仅保留“成交/未成交”。
- 公海规则：可配置规则集合，决定客户何时退公海、如何认领以及何时提醒负责人。
- 用户与角色：系统管理员、销售主管、销售专员、只读或辅助角色。
- 外部映射：用于关联 ERP 编号、渠道账号和后续同步状态。

## 核心组件

- 接口层：
  - 线索接入入口，包括 WhatsApp、邮箱、网站聊天和手工录入
  - 客户列表、详情页、联系人页、商机页和系统设置页
- 领域逻辑：
  - 联系人建档与客户归属
  - 跟进事件落库与时间字段更新
  - 公海规则评估、认领、分配和转移
  - 轻量商机状态维护
- 持久化或外部集成：
  - 关系型数据存储，用于客户、联系人、事件、规则和权限
  - 邮件接入、网站聊天接入、WhatsApp 接入
  - ERP 编号和后续同步适配层
  - 附件和文档存储，首版按需落地为 S3 兼容对象存储
- 后台任务或定时任务：
  - 公海自动退回规则评估
  - 提醒任务生成
  - 渠道同步与消息补录

## 核心规则

- 未建档联系人不受归属保护，任何业务员都可继续跟进。
- 联系人转为客户后，客户进入归属与公海规则体系。
- 跟进行为由配置决定，配置项触发后更新最近跟进时间。
- 客户最近获得时间、最近跟进时间、最近成交时间均为系统计算字段，不依赖人工直接维护。
- 公海规则必须支持配置，不得将单一规则写死在实现中。
- 当前首个规则内核放在 `twenty-shared` 共享层，而不是前端页面层：
  - `computeCrmCompanyPoolState` 负责根据时间字段与规则配置计算 `PUBLIC_POOL` 迁移、预计退公海时间和提前提醒。
  - `getCrmPersonProtectionState` 负责根据 `company` 关联和 `crmOwnershipStatus` 判断联系人是否受保护。
- 当前首个服务端适配层放在 `twenty-server` 的 `engine/core-modules/company-public-pool/`：
  - `CompanyPublicPoolService` 负责将共享规则输出收敛成最小 Company patch。
  - 当前先不直接绑定 cron、job 或写入 hook，先确保服务层可独立复用和测试。

## 核心流程

1. 线索进入系统
   - 来自 WhatsApp、邮箱、网站聊天或手工录入的触点进入 CRM。
2. 触点识别与建档
   - 触点以未建档联系人存在，后续可转为正式客户。
3. 跟进行为沉淀
   - 发/收消息、发/收邮件、手工记录和新建商机被记录为事件。
4. 时间字段更新
   - 规则引擎根据事件更新最近获得时间或最近跟进时间。
5. 公海规则评估
   - 定时任务评估客户是否满足退公海、认领或提醒条件。
6. 商机状态变更
   - 商机从未成交切换为成交时，系统更新客户最近成交时间。

## 集成点

- 认证与授权：
  - 内部账号体系与角色权限控制
  - 是否保留邮箱/手机号登录方式待后续技术选型确认
- 数据存储：
  - 支持客户、联系人、跟进事件、公海规则、角色权限和 ERP 映射的关系型存储
- 队列或 worker 系统：
  - 支撑定时规则评估、提醒任务和渠道同步任务
- 外部 API 或内部服务：
  - WhatsApp 接入
  - 邮件收发与同步
  - 网站聊天或表单接入
  - ERP 同步接口
  - 阿里国际站和微信相关能力先预留边界，不作为首版强依赖

## 安全与可靠性约束

- 外部渠道账号、登录状态、消息内容和客户资料都视为敏感数据，必须限定权限访问。
- 涉及归属、公海、分配和转移的关键操作需要保留审计记录。
- 在未明确生产边界前，不默认对真实渠道账号和生产 ERP 执行写操作。
- 公海规则和批量操作必须支持可追溯，以便在规则配置错误时快速恢复。

## 待确认问题

- 与 Twenty 的集成方式尚未最终确认，是 fork 跟随上游还是一次性导入后独立维护。
- ERP 第一阶段同步的数据粒度尚未确认，是仅保留 ERP 编号，还是同步部分成交结果。
- 网站聊天的具体接入方式和现有供应商尚未确认。
- 附件与沟通文件是否在首版落地为对象存储尚未确认。
