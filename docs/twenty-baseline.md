# Twenty 基线决策

## 决策摘要

- 本仓库放弃继续推进此前的自定义绿地框架方案。
- 本仓库改为采用 [Twenty](https://github.com/twentyhq/twenty) 作为上游 CRM 基线。
- 后续实现优先遵循 Twenty 的工程结构、模块边界、命名方式、构建脚本和测试约定。
- 本仓库只围绕已确认的前台 CRM 范围做裁剪与扩展，不再平行维护一套独立框架。

## 选择原因

- Twenty 已经是成熟的开源 CRM，具备客户对象、权限、活动流、工作流等通用 CRM 能力。
- 对本仓库而言，采用成熟基线比从零搭框架更有利于控制范围、提高实现速度并降低规范发散。
- 对 Codex 而言，已有上游代码和既定约定比空白仓库更容易持续开发和保持一致性。

## 本仓库保留的业务目标

- 前台 CRM 定位不变。
- 线索来源仍以 WhatsApp、邮箱、网站聊天和手工录入为首批范围。
- 未建档联系人不受归属保护。
- 商机首版仍只保留“成交/未成交”。
- 公海规则仍以询盘云为参考，并保持可配置。
- ERP 仍负责成交后的订单和财务事实。

## 本仓库的主要裁剪点

- 不引入超出前台 CRM 边界的 ERP 能力。
- 不优先开发复杂营销自动化、大屏 BI 和第二阶段 AI 能力。
- 不为了贴合旧方案而强行保留此前自定义的 Next.js + NestJS 绿地架构。

## 实施原则

- 先复用 Twenty 现有能力，再决定是否需要本地覆盖。
- 本地新增模块必须说明为何不能直接复用上游。
- 本地定制尽量保持低分叉，降低后续上游同步成本。
- 若上游已存在等价能力，优先用配置、扩展点或薄层适配解决，而不是重写。
- 当前第一批 CRM 对象差异采用 `Company` / `Person` 的 custom field metadata 流落地，而不是在当前导入分支中直接改缺失的标准对象源码。
- 当前第一批业务规则也先放入共享层 `twenty-shared/src/utils/crm/`，避免把公海到期和联系人保护绑定到 `twenty-front` 页面状态。

## 待确认事项

- 采用方式是直接 fork 并长期跟随上游，还是一次性导入后独立维护。
- 本仓库与上游的同步节奏、升级窗口和冲突处理方式。
- 哪些上游模块直接启用，哪些模块需要在首版禁用或隐藏。
