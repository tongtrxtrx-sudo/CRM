# 简报：CRM

## 问题

- 现有客户线索分散在 WhatsApp、邮箱、网站聊天和手工记录中，销售跟进信息难以统一沉淀。
- 未建档联系人、已建档客户、公海客户之间边界不清时，容易出现抢客、漏跟进和客户闲置。
- 当前 CRM 类工具与 ERP 分离运行，成交前后的信息链路断裂，导致重复录入和统计失真。
- 业务当前需要的是前台销售 CRM，而不是重做一套覆盖财务、库存和履约的 ERP。

## 用户或操作者

- 主要：外贸销售专员、销售主管
- 次要：系统管理员、销售支持或只读查看角色

## 期望结果

- 将 WhatsApp、邮箱、网站聊天和手工录入的线索统一进入一个前台 CRM。
- 让联系人建档、客户归属、公海流转和跟进提醒有一致且可配置的规则。
- 让销售主管能够看到最近获得时间、最近跟进时间、最近成交时间和预计退公海日期。
- 将成交前客户关系管理保留在 CRM 中，成交后的订单与财务事实继续留在 ERP 中。

## 范围内

- 线索接入：WhatsApp、邮箱、网站聊天、手工录入
- 客户与联系人管理：客户以公司为主体，支持一个客户下多个联系人
- 建档与归属：未建档联系人转为客户后才进入归属保护
- 跟进事件沉淀：发/收 WhatsApp、发/收邮件、网站聊天、手工跟进记录、新建商机
- 时间字段维护：最近获得时间、最近跟进时间、最近成交时间
- 公海规则：参考询盘云，支持可配置的自动退回、认领和提醒
- 轻量商机：仅保留“成交/未成交”状态
- 权限模型：系统管理员、销售主管、销售专员、只读或辅助角色
- ERP 关联：保留 ERP 编号和后续同步接口，不在 MVP 中承接 ERP 业务流程

## 范围外

- 财务、库存、采购、发货、开票等 ERP 能力
- 复杂销售漏斗、完整跟单流程和多阶段履约编排
- 强依赖阿里国际站、微信等未稳定开放接口的深度集成
- AI 自动外呼、客户背调、智能话术和培训助手，这些放到第二阶段
- 大屏 BI、复杂经营分析和营销自动化平台级能力

## 约束

- 产品边界必须保持为前台 CRM，不得在技术设计阶段滑向 ERP 一体化。
- 公海规则必须可配置，不能将询盘云当前规则硬编码为唯一方案。
- 未建档联系人不受归属保护，任何业务员都可继续跟进。
- 技术栈先锁定为 TypeScript 单仓方案：Next.js Web、NestJS API、PostgreSQL、Redis 和 BullMQ。
- 在权限边界、客户数据边界和外部系统边界未完全明晰前，不默认接入任何生产系统。

## 技术选型

- 仓库形态：`pnpm workspace` 单仓，多应用目录至少包含 Web、API 和 Worker。
- Web：Next.js App Router + React + TypeScript，用于 CRM 工作台、列表页、详情页和系统设置页。
- UI：Tailwind CSS + `shadcn/ui` + TanStack Table，用于数据密集型后台界面。
- API：NestJS + Fastify，暴露 REST API，并预留 OpenAPI 文档能力。
- 数据：PostgreSQL 作为主存储，核心字段结构化建模，外部渠道原始载荷按需使用 `jsonb`。
- 异步任务：Redis + BullMQ，负责公海扫描、提醒、渠道同步和后续集成任务。
- ORM 与迁移：Drizzle ORM + drizzle-kit。
- 测试：Vitest 用于单元与服务层测试，Playwright 用于关键流程 E2E。
- 部署：本地开发和首版部署均以 Docker Compose 为默认目标。

## 成功信号

- 销售能够在一个系统内查看四类来源的线索和统一跟进历史。
- 销售主管能够通过规则配置控制客户退公海、认领和提醒。
- 客户列表可按负责人、公海、国家、客户编号、ERP 编号、最近跟进时间等字段筛选。
- 成交前客户管理不再依赖人工散落记录，重复建档和漏跟进明显减少。

## 验收标准

- 场景：多渠道线索进入系统
  假设 销售收到 WhatsApp、邮箱或网站聊天的新触点
  当 系统接收或手工录入该触点
  则 该触点以联系人或未建档线索形式进入 CRM，并可被后续建档

- 场景：联系人建档并归属
  假设 某触点尚未建档
  当 销售将其转为客户
  则 系统创建客户主体、保留联系人信息并建立负责人归属

- 场景：跟进行为更新最近跟进时间
  假设 客户已存在
  当 销售发生被配置为有效跟进的行为
  则 系统更新最近跟进时间并记录活动明细

- 场景：客户自动退公海
  假设 客户满足某条已启用的公海规则
  当 定时规则评估执行
  则 客户自动退入公海，并在设定提醒天数内提前通知负责人

- 场景：商机标记成交
  假设 客户存在一条商机
  当 销售将该商机标记为成交
  则 系统更新客户最近成交时间，并保持状态仅为“成交/未成交”

## 风险与假设

- 风险：若后续又将订单、库存或财务诉求纳入本项目，会显著扩大范围并拖慢落地。
- 风险：邮箱与社媒消息中存在大量非销售线索内容，若缺少分类策略会污染工作台。
- 风险：外部渠道接入能力差异较大，部分渠道早期只能通过手工或半自动方式接入。
- 假设：第一阶段先以 CRM 管理成交前客户关系，ERP 仍为成交后业务事实源。
- 假设：公海规则以询盘云为参考模型，但最终规则集由本项目配置和维护。
