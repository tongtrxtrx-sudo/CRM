## 问题

- 为外贸销售团队提供一套前台 CRM，用于统一管理 WhatsApp、邮箱、网站聊天和手工录入的线索、客户、联系人、跟进记录和公海规则。
- 当前系统需求集中在成交前客户关系管理，而不是重做覆盖财务和履约的 ERP。

## 范围

- 范围内：
  - 四类来源的线索接入：WhatsApp、邮箱、网站聊天、手工录入
  - 客户与联系人管理，支持未建档联系人转为客户
  - 跟进事件记录与最近获得时间、最近跟进时间、最近成交时间维护
  - 可配置公海规则、认领、分配、转移和提醒
  - 轻量商机，状态仅为“成交/未成交”
  - 角色权限与基础组织架构
  - ERP 编号与后续同步接口预留
- 范围外：
  - 财务、库存、采购、发货、开票等 ERP 能力
  - 复杂销售漏斗、完整跟单编排和多阶段履约状态
  - AI 自动外呼、客户背调、话术助手和培训助手
  - 强依赖阿里国际站、微信等未稳定开放接口的深度集成

## 实现约定

- 单仓与包管理：`pnpm workspace`
- Web：Next.js App Router + React + TypeScript
- API：NestJS + Fastify + REST
- 数据库：PostgreSQL
- ORM 与迁移：Drizzle ORM + drizzle-kit
- 队列与定时任务：Redis + BullMQ
- 测试：Vitest + Playwright
- 本地和首版部署：Docker Compose
- 首版不引入第三方 Auth SaaS，采用自管账号体系与角色权限模型

## 需求

### 功能需求

- FR-1：系统必须支持通过 WhatsApp、邮箱、网站聊天和手工录入创建线索或触点联系人。
- FR-2：系统必须支持将未建档联系人转为正式客户，并在客户主体下维护多个联系人。
- FR-3：系统必须记录有效跟进行为，并根据配置更新最近获得时间、最近跟进时间和最近成交时间。
- FR-4：系统必须支持可配置的公海规则，至少覆盖自动退回、认领、分配、转移和提醒。
- FR-5：系统必须允许未建档联系人不受归属保护，任何业务员均可继续跟进。
- FR-6：系统必须提供轻量商机能力，并将商机状态限定为“成交/未成交”。
- FR-7：系统必须支持按负责人、公海、国家、客户编号、ERP 编号、最近时间字段等条件检索客户。
- FR-8：系统必须提供角色权限控制，至少覆盖系统管理员、销售主管、销售专员和只读或辅助角色。

### 非功能需求

- NFR-1：公海规则和有效跟进行为必须通过配置维护，不依赖代码变更。
- NFR-2：关键操作必须具备可审计性，至少覆盖建档、分配、转移、退公海和商机状态变更。
- NFR-3：系统设计必须保持前台 CRM 边界，不得在首版中承担 ERP 业务事实。
- NFR-4：在未确认生产边界前，所有真实外部账号和真实客户数据接入都必须经过显式确认。

## 核心流程

1. 线索进入与建档
   - 触点从 WhatsApp、邮箱、网站聊天或手工录入进入系统。
   - 触点以未建档联系人存在，后续可转为客户。
2. 跟进沉淀与时间更新
   - 跟进行为落为事件。
   - 系统根据配置更新最近获得时间或最近跟进时间。
3. 公海规则评估
   - 定时任务评估客户是否满足退公海条件。
   - 系统在配置的提醒天数内提前通知负责人。
4. 成交确认
   - 销售将商机状态改为成交。
   - 系统更新客户最近成交时间。

## 验收标准

- 场景：创建未建档联系人
  假设 销售收到新的 WhatsApp、邮件或网站聊天触点
  当 该触点进入系统
  则 系统创建未建档联系人，并允许后续建档

- 场景：未建档联系人可被任意业务员继续跟进
  假设 某联系人尚未转为客户
  当 其他业务员查看并继续跟进该联系人
  则 系统允许该行为，不应用归属保护

- 场景：客户自动退公海
  假设 某客户满足已启用的退公海规则
  当 定时规则评估执行
  则 系统将客户退入公海，并保留可审计记录

- 场景：商机标记成交
  假设 客户存在一条商机
  当 销售将该商机标记为成交
  则 系统将客户最近成交时间更新为该商机成交时间

## 风险与假设

- 风险：若后续将 ERP 诉求重新并入 MVP，范围会显著膨胀并拖慢交付。
- 风险：邮件与社媒中混有大量非销售内容，若无后续分类策略会降低系统可用性。
- 风险：部分渠道的集成方式在首版只能做到手工或半自动接入。
- 假设：ERP 继续负责成交后的订单和财务事实，CRM 只负责成交前客户关系。
- 假设：AI 相关能力均放在第二阶段，不进入首版交付。
